## What Was Built

### Net Worth Leaderboard

- Batch computation of player net worth: `balance + portfolio value`
- Portfolio includes gate shares (market price or fundamental fallback) and guild shares (market price or creation cost fallback)
- Collapsed/dissolved assets excluded from valuation
- Treasury-held and guild-self-held shares excluded from player portfolios
- Reuses `_share_value_micro` from anti-exploit for consistent valuation

### Activity-Based Score Decay

- Score derived from net worth with inactivity penalty
- Grace period of 100 ticks before decay starts
- Linear decay at 0.01% per inactive tick beyond grace
- Floor at 50% of net worth — returning players never fully wiped
- Activity detected from executed intents and placed orders

### Seasonal Competition

- Automatic season creation on first tick
- Season rotation when duration exceeded (default: 17,280 ticks ≈ 1 day)
- Season finalization records ranked standings for all non-AI players
- Historical season results preserved with frozen ranks and scores
- Concurrent-safe season creation via savepoints (handles worker race)

### API Endpoints

| Method | Path               | Auth | Purpose                                 |
| ------ | ------------------ | ---- | --------------------------------------- |
| GET    | `/leaderboard`     | No   | Paginated rankings (non-AI players)     |
| GET    | `/leaderboard/me`  | Yes  | Authenticated player's rank + breakdown |
| GET    | `/seasons`         | No   | List all seasons (paginated)            |
| GET    | `/seasons/current` | No   | Current active season                   |

## Design Decisions

| Decision                                      | Rationale                                                     |
| --------------------------------------------- | ------------------------------------------------------------- |
| Net worth = balance + portfolio               | Escrow already deducted from balance; conservative            |
| Decay based on activity, not time             | Rewards participation, not passive holding                    |
| AI excluded from API, included in computation | Data completeness; filtered at presentation layer             |
| Leaderboard updated every N ticks             | Performance: batch computation expensive with many players    |
| Season checked every tick                     | Time-critical transitions can't lag behind update interval    |
| Season creation via savepoints                | Handles concurrent tick execution (worker + test)             |
| No rewards this phase                         | Simple foundation; rewards can be added later with new faucet |
| Rank computed at query time                   | Avoids recomputing all ranks on every score change            |
| Rank stored in season_results                 | Frozen at season end for historical accuracy                  |

## Infrastructure Fix: Worker-Test Isolation

Resolved a pre-existing class of flaky test failures caused by the Celery worker running ticks concurrently with tests:

- **Root cause:** Tests creating `Tick(tick_number=1)` raced with worker's `execute_tick` creating the same tick
- **Fix:** Made `pause_simulation` (Redis lock acquisition) a dependency of the autouse `_clean_state` fixture, so all tests hold the simulation lock throughout execution
- **Lock tests:** Given unique lock keys via optional `lock_key` parameter to avoid interference with global pause
- **Result:** Eliminated all intermittent `UniqueViolationError` failures across the entire test suite
